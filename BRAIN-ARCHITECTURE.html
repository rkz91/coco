CoCo Brain Architecture

Current state (scattered) → Target state (unified)

1. Current State — "The Sprawl"

Problems with current state

  • 5 separate storage locations — ~/.coco, ~/.hub, ~/.claude/skills, ~/.claude/media-memory, ~/.claude/projects/memory
  • 4 overlapping databases — coco.db, platform.db, knowledge.db, hub.db all store entities/sessions differently
  • 3 memory systems — brain.json (people), mempalace (vectors), auto-memory (markdown) doing similar things
  • Broken cron — master-cron exits with "Too many open files", wiki generation stalled
  • 277MB dead weight — mempalace-env venv sitting in data directory
  • No git, no backup — one disk failure loses everything
  • queue.json at 313 pending — items accumulating but never draining
  • Wiki half-built — 545 entity articles, but 0 topic articles, 0 relationship articles

2. Target State — "Unified Brain"

What changes

  • 1 database instead of 4 — coco.db becomes the single source of truth (migrate platform.db + knowledge.db tables in)
  • 1 cron job instead of 2 — unified processor with proper fd limits fixes the "Too many open files" crash
  • Code in git, data in ~/.coco/ — clean separation. Code is versioned, data is backed up
  • brain.json → DB table — people graph moves to SQLite (queryable, no more atomic JSON dance)
  • queue.json → DB table — 313 pending items become a proper queue with indexes
  • events.jsonl → DB table — 17K events become queryable instead of grep-only
  • Kill mempalace-env — move ChromaDB to platform's venv, delete 277MB dead weight
  • Kill memory-bus — replaced by DB-backed event stream
  • Wiki completes — fix cron, generate topic + relationship articles
  • Everything pushable — code to GitHub, data backed up via scheduled DB dump
